Decision Brief

How GIS data moves through QAViewer today, and where the architecture should change.

QAViewer currently works as a Dockerized GIS review prototype. It takes a prepared geodatabase, exports normalized GeoJSON seed files, imports those files into PostGIS, and serves map-ready GeoJSON to a Leaflet interface. That path is workable for a prototype, but it blurs the difference between source GIS data, derived question areas, and review workflow state.

557 question areas loaded into PostGIS
3,373 primary parcel polygons
3,079 parcel point features
896 management tract polygons
Current State

What the app is using right now

The browser does not open the geodatabase directly. It asks the API for map features. The API reads PostGIS tables, converts geometry to GeoJSON, and sends that GeoJSON to Leaflet.

This keeps PostGIS safely behind the API, but it also means the correctness of the review experience depends heavily on the seed export step.

The current loaded question areas appear to come from BTG_Spatial_Fix_Primary_Layer, which is the primary parcel context layer. The architecture guide says question areas should come from the mismatch erase layers. That is the biggest data concern found during this review.

Review Findings

What needs attention before the app becomes a dependable review system

  • Question-area source looks wrong. Loaded question areas report BTG_Spatial_Fix_Primary_Layer as the source layer. The intended source layers are the primary and comparison erase mismatch layers.
  • Comparison mismatch data is not represented in the current snapshot. The manifest shows a primary breakdown only, so reviewers may be seeing one side of the problem.
  • Question-area geometry is not visually emphasized. The app fetches question-area polygons but renders them transparent, leaving question mark markers as the main visual signal.
  • Parcel relationships are implicit. The app links question areas to parcels using matching keys at query time instead of storing a clear relationship record with match type and confidence.
  • There are coordinate outliers. 17 parcel points fall outside a normal US coordinate envelope. That points to source data cleanup or validation work.

The core application stack is still reasonable: Docker, Express, PostGIS, and Leaflet are a good fit. The recommended change is not to throw away the stack. The recommended change is to formalize the GIS import model and separate source data from derived review records.

Diagram 3

Recommended architecture

The strongest next architecture adds an import/staging layer, derives question areas from explicit mismatch sources, and stores relationships between question areas and related GIS features.

Preserve lineage. Every feature should keep dataset, import run, source layer, source feature ID, and raw attributes.

Derive review objects. Question areas should be generated from explicit mismatch layers or a repeatable PostGIS process.

Store relationships. Related parcels, points, and management tracts should be recorded with relationship type and confidence.

Decision Points

Questions worth answering before the next build

1. What is a question area?

Current answer: a question area is a location where property tax boundaries may not match legal deed retracement or the management/ownership data the client uses to represent what they own and manage. The schema and UI should preserve that distinction instead of treating it as a generic parcel record.

2. Which layer is authoritative?

The app needs a clear vocabulary for primary, comparison, management, parcel, and point layers. Reviewers should be able to tell which side produced the mismatch.

3. Should import be repeatable?

If the same source data is imported twice, the system should produce the same question areas or explain what changed. That requires import runs, hashes, and validation reports.

4. What should users see first?

The map should highlight the actual mismatch polygon, then show related parcel and management geometry as context. Markers should help navigation, not replace the spatial evidence.

Recommended next steps

  • Regenerate seed data and confirm question areas come from the two mismatch erase layers.
  • Add importer validation for missing comparison output, wrong source layer, and coordinate outliers.
  • Update the map styling so question-area polygons are visible and selected mismatch geometry is prominent.
  • Design the next database model around datasets, import runs, source features, derived question areas, and explicit relationships.
